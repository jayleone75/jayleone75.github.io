◆ Case Study 01

Public-Sector Procurement Intelligence Dashboard

Enterprise Sales Tool HTML / JS / PowerShell ~450 records mapped

The problem

California publishes state-agency procurement data, but it lives scattered across separate spreadsheets — one per vendor, with inconsistent schemas, mixed statuses, and no way to see the same department's buying pattern across competitors. For a seller working that territory, the data was technically public and practically unusable.

What I built

A single-page dashboard that ingests four competitor spreadsheets, normalizes them into a unified JSON dataset, and renders a live filterable view of the combined market. Keyword search across items, suppliers, buyers, departments, and POs. Faceted filters for source, status, department, and supplier. Live summary metrics for spend, deal size, and active/closed mix. Rollup panels for top suppliers, top departments, and source distribution.

The interesting part

The dashboard was the easy half. The harder problem was turning data into action — so I layered an intelligence engine on top: a curated lifecycle dataset of product SKUs with end-of-support dates, confidence tiers, and source URLs. Past-EOS hardware now surfaces automatically as the top prospecting angle, with specific model names, PO counts, and dates — and the tool exports a ready-to-use briefing document in Markdown or styled HTML that feeds directly into Claude for draft emails, partner briefs, and account strategy.

The AI-native piece

This is where the tool stops being a dashboard and starts being a genuine force multiplier. The brief export is designed as a prompt payload — structured so I can paste it into Claude with one line of intent ("draft a prospecting email to this buyer") and get a useful first draft in seconds. The whole workflow — data → filter → brief → email — collapses from hours to minutes.

◆ Case Study 02

IronLog — A Workout Tracker, Built in the Browser

Personal · Fitness Progressive Web App Single-file HTML · ~1,200 lines ↗ Live Site

The problem

Existing fitness apps don't track the specific style of high-intensity training I follow — minimal sets, maximal effort, careful tracking of progressive overload across a four-day split. I also didn't want another account, another subscription, another app store dependency. I wanted something I could open on my phone in the gym and trust to remember everything.

What I built

A mobile-first Progressive Web App that runs entirely in the browser — no backend, no app store, no account. It installs to the Android home screen from a GitHub Pages URL and behaves like a native app. The full product includes a four-day split tracker pre-loaded with specific exercises, a rest timer, a workout history log, and an AI-powered insights engine — all in a single self-contained HTML file with zero dependencies and zero build step.

The interesting technical challenges

Mobile storage reliability. The first version, opened from a file manager, caused Android Chrome to treat each session as a new origin — wiping localStorage every time. The fix was hosting at a stable URL. A seemingly trivial problem that required understanding how mobile browsers actually scope storage.

Autosave and crash recovery. I kept losing workouts by forgetting to tap "Finish." The fix was a dual-key storage architecture: completed sessions write to one key, while an in-progress session continuously autosaves to a second key on every input event. On page load, the app checks for orphaned autosaves, de-dupes against completed sessions, and offers a one-tap recovery prompt if unfinished work is found.

Last-session pre-population. Rather than starting blank, every workout pre-fills with the most recent session's exact weights, reps, and notes for the same split — with the prior notes flagged in orange so they're impossible to miss (visible in the Workout screenshot above). Required a clean history data model keyed by split and exercise name.

The AI-native piece

The Insights engine runs entirely client-side with no API calls — detecting personal records, flagging stalls, calculating volume trends, surfacing contextual coaching tips. A separate "Analyze with Claude" button formats the full workout history into a structured prompt, copies it to the clipboard, and opens Claude.ai — giving deep on-demand analysis without requiring an API key in the app itself. Cheap, simple, and surprisingly powerful.

◆ Case Study 03

The Leone Family Adventure — A Road Trip Companion

Personal · Family Real-time Web App 9 days · 3 states · 5 people ↗ Live Site

The problem

Planning a 9-day family road trip across California, Arizona, and Utah meant juggling itinerary, reservations, drive times, kids' attention spans, photo coordination across five phones, and a partner who reasonably wanted to know where the Sprinter van was at any given moment. Existing tools handled parts of that — but nothing handled the whole job for a specific family on a specific trip with specific quirks.

What I built

A custom single-page web application — roadtrip.jayleone.ai — designed to plan, navigate, and capture the trip as it happened. Five tabs: Schedule (day-by-day itinerary with drive times, stops, reservations), Live Map (real-time GPS tracking of the van pushed from my phone), Overview (trip stats and big-picture context), Checklist (packing and pre-departure tasks), and Photos (on-the-road uploads from everyone's phones, pinned to locations). Designed to work seamlessly from the passenger seat of a moving vehicle on spotty rural cell signal.

The interesting technical challenges

Live GPS tracking in a browser. Browsers aggressively sleep the Geolocation API when a tab backgrounds or signal drops. After evaluating heavier self-hosted solutions, I built a purpose-made gps.html page that stays foregrounded, polls location on an interval, and writes coordinates directly to Supabase. The Live Map tab renders the result with a moving van icon — so my wife and kids could watch the van's progress in real time from the back seat.

Photo uploads under real conditions. Uploading from a moving vehicle on bad 3G is its own discipline. I used direct-to-Supabase Storage uploads with client-side compression and graceful retry behavior. Added per-photo delete controls using event delegation on data attributes — critical after one family member uploaded forty near-duplicates of the same canyon.

The auto-refresh bug that wiped uploads in progress. Early Photos tab refreshed every 30 seconds to pull new images. Unfortunately, that also wiped the in-progress file picker state mid-upload. Disabled auto-refresh on that tab specifically and added a manual button. Small bug, large UX impact.

Offline-ish operation. Much of the route had no signal — Grand Canyon rim, Zion back roads, long stretches of Utah. The app degrades gracefully: cached itinerary, checklist works without backend, photos queue locally until reconnection.

Schedule reflow when delayed. If we left a stop late, the app recalculated the day — surfacing how much time we'd lost, which downstream stops were at risk, and what we might want to drop or shorten (the "Running late? Tap to recalculate" control is visible in the Schedule screenshot above).

The AI-native piece

The whole app — frontend, backend integration, real-time sync, geolocation handling, deployment scripts — was built through natural-language conversation with Claude. What's noteworthy isn't that AI wrote the code. It's that the architecture decisions (Supabase over self-hosting, dedicated GPS page over service workers, no auth over a login system) all came from real tradeoffs I could reason about and Claude could implement. The build felt less like coding and more like leading a very fast, very competent contractor.

What it taught me Most consumer software is built for a generic user and optimized for ad revenue. This one was built for exactly five people, had zero ads, zero logins, and was tuned to the realities of a specific van, a specific route, and specific phones. Building software for a known audience of five is a different kind of fun than building for millions — and it's the part of the AI-native shift most people haven't internalized yet. The economics of "software built for one family" used to be zero. They aren't anymore.
